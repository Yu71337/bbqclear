# 多语言纯前端编辑器项目设计方案 (Multi-Language XLIFF Editor)

## 1. 概述与背景
该项目旨在构建一个本地独立的新项目应用（纯前端 SPA），提供 Crowdin XLIFF 语言包文件的解析、浏览、搜索和在线修改功能。
项目核心亮点在于整合了大语言模型 (LLM)，使其在每个字符串词条处提供基于规则的智能翻译建议，减轻多语言校对和翻译的工作量。

## 2. 架构设计
- **技术栈**: React 19 + Vite + TypeScript.
- **UI 组件**: Tailwind CSS + Lucide React + 无头组件库 (如 shadcn/ui 或原生 HTML 复合组件)
- **核心文件解析**: 采用 `fast-xml-parser`，支持 XML 和 JSON 的双向高性能转换。
- **状态存储**: 
  - `localStorage` 持久化大模型配置（API Key, Base URL, Model Name）。
  - Context / Zustand 存放当前载入的全局文件对象状态及搜索过滤条件。
- **环境部署**: 本地开发或一键构建部署为浏览器应用。所有操作依赖浏览器前端内存运算及 Fetch API，无自身后端。

## 3. 核心可视化界面设计

### 3.1 全局组件
- **顶部操作栏**: 
  - **导入按钮**: 支持上传 `.xliff` 格式文件。
  - **导出按钮**: 将修改后的内容实时转换为 `.xliff` 并触发浏览器下载。
  - **设置弹窗 (Global Settings)**: 输入大模型 API 的 Endpoint、Model 和 Token。
- **检索与操作栏 (Filter Bar)**: 
  - ID 模糊检索
  - 源码端 (Source) 模糊检索
  - 目的端 (Target) 模糊检索
  - “仅查看未翻译 / Needs-Translation状态” 勾选框
  
### 3.2 翻译工作台列表
从上至下平滑渲染条目卡片，每张卡片含：
- **元数据信息 (左侧)**: 
  - 唯一标识 `ID`
  - 翻译状态 `State` (如下拉 `needs-translation` / `translated` 等)，由于与 `target` 节点绑定，支持双向更新和可视化选择。
  - `Source Text` 显示原文，并高亮参数或占位符 (如 `{name}`)。
- **实时编辑区 (右侧)**: 
  - `Target Text` 供使用者实时手动敲击和修改的 Textarea，并支持 `onBlur` 随时写回内存状态。
- **智能辅助区 (底部/交互区)**:
  - 提供 “获取 AI 建议” 按钮触发智能翻译请求。

## 4. 大模型(LLM)智能建议机制

### 4.1 通讯规范
依靠 OpenAI 兼容格式 (`messages: [{role: 'user', content: ...}]`) 与使用者所配置的兼容接口地址发起原生 Fetch 调用。

### 4.2 翻译规则 (提示词工程约束)
提取文件头部指定的 `<target-language>` 得到目标语种后，应用以下系统提示词注入与解析：
1. **短语保留规则 (Localization Optimization)**: 
   针对 App 中常见的短语（特别是按键如 "OK"），无需机械直译，若该语种常作保留英文处理，或有约定俗成的短语，请直接使用该约定。
2. **防溢出长度约束 (Length Constraint)**: 
   受限于 UI 界面边框大小，输出翻译后的视觉长度（或字符长度）必须“尽可能趋近原文或小于原文”。宁可使用标准缩写也不可冗长。
3. **占位符保护机制 (Parameter Skipping)**: 
   明确标记源句中的参数（以 `{}` 或其他通配符出现的部分，例如 `{name}`），指示大语言模型跳过对其变量名的翻译，必须严格原样放置到译文合适的位置，不可带入大小写变更。

### 4.3 确认逻辑
大模型返回流式呈现或者完毕后展示给用户。在 AI 提供区提供 "一键采用" 按钮；点击后，会将内容写入本地 `Target Text` 区，并将条目的 `State` 更新为 "translated"。
